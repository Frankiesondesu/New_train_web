# 网站改进清单

| 优先级 | 模块 | 需要补充的内容 | 为什么需要 | 建议做法 | 状态 |
| --- | --- | --- | --- | --- | --- |
| 高 | 数据 | 接入真实铁路线路、车站、车次和时刻数据 | 当前内容更像演示数据，正式网站需要可信、可更新的数据来源 | 先定义统一的数据结构，再接入官方或授权 API，也可以先用可维护的 JSON 数据文件过渡 | 已完成第一版 |
| 高 | 内容 | 修复页面标题、描述和 README 的中文乱码风险 | 如果部署后出现乱码，会直接影响搜索、分享和用户信任 | 统一保存为 UTF-8，检查 HTML meta、编辑器编码和 GitHub 展示效果 | 已完成 |
| 高 | 地图 | 增加地图加载失败时的明确提示和重试按钮 | 当前已有底图降级思路，但用户需要知道发生了什么以及如何恢复 | 在地图区域展示轻量错误状态，提供重试和切换底图入口 | 已完成 |
| 高 | 移动端 | 优化小屏下搜索、筛选、详情面板的层级关系 | 旅行地图主要会在手机上使用，面板遮挡和操作路径会影响体验 | 做一次 375px、390px、430px 宽度的交互检查，固定关键控件尺寸和层级 | 已完成 |
| 中 | 搜索 | 支持拼音、城市名、景点名和模糊搜索 | 用户不一定知道完整站名或线路名 | 为站点和线路增加 aliases 字段，搜索时匹配别名、拼音和关键词 | 已完成 |
| 中 | 收藏 | 做成真正可用的收藏功能 | 导航里已有“我的收藏”，但需要可保存站点、线路和行程 | 用 localStorage 保存收藏列表，后续再接账号系统 | 已完成 |
| 中 | 行程 | 增加“加入行程”或“生成旅行路线”功能 | 网站主题是铁路旅行，用户需要从浏览转向规划 | 支持选择出发站、目的站、停留天数和偏好，生成建议路线 | 待做 |
| 中 | 详情 | 丰富车站周边信息的来源和可操作性 | 景点、住宿、风景信息如果不可验证，长期维护成本会高 | 给每条内容增加来源、更新时间、外链和标签 | 待做 |
| 中 | 性能 | 把路线、站点和内容数据拆出 `app.js` | 当前 JS 同时包含数据和交互逻辑，后续会越来越难维护 | 新增 `data/routes.js` 或 JSON 文件，把渲染逻辑和数据分离 | 已完成 |
| 中 | SEO | 补充正确的页面标题、描述、Open Graph 和站点图标 | 方便搜索引擎收录，也让分享链接更清晰 | 添加 `og:title`、`og:description`、`og:image`、favicon 和 manifest | 待做 |
| 中 | 可访问性 | 完善键盘操作、焦点样式和 ARIA 状态 | 地图和弹层类页面容易让键盘用户迷失 | 检查 Tab 顺序、按钮名称、弹层焦点管理和对比度 | 待做 |
| 低 | 视觉 | 增加真实铁路、城市或风景图片资产 | 当前体验偏工具型，适量真实图片能增强旅行感 | 使用授权图片，按线路或站点配置封面图和缩略图 | 待做 |
| 低 | 分享 | 增加站点、线路详情的可分享链接 | 用户可以把某条线路或某个车站发给别人 | 使用 URL hash 或 query 参数记录当前选中对象 | 待做 |
| 低 | 测试 | 增加基础自动化检查 | 后续改动地图、搜索和移动端时更容易发现回归 | 添加轻量 Playwright 冒烟测试，检查页面加载、搜索、筛选和主题切换 | 待做 |
| 低 | 部署 | 配置 GitHub Pages 或其他静态部署流程 | 上传到 GitHub 后还需要一个稳定的公网访问地址 | 使用 GitHub Pages 部署静态文件，README 里补充访问链接 | 已完成 |

## 建议先做的 5 件事

1. 扩展更多线路和站点，提升真实数据覆盖范围。
2. 增加自动更新数据的脚本或后端接口。
3. 继续扩展真实数据覆盖范围，增加更多线路、停靠站和可更新来源。
4. 做可用的收藏功能，让已有导航按钮变成真实功能。
5. 增加移动端细节检查，保证地图、筛选和详情面板不会互相遮挡。
